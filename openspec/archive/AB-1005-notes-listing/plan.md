## Prisma Schema Changes

```prisma
model Note {
  id        String    @id @default(cuid())
  userId    String
  title     String
  body      Json
  bodyText  String    @default("")
  version   Int       @default(1)
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  deletedAt DateTime?

  user     User          @relation(fields: [userId], references: [id], onDelete: Cascade)
  versions NoteVersion[]

  @@index([userId, deletedAt])
  @@index([userId, createdAt])
  @@index([userId, updatedAt])   // NEW
}
```

- Purely additive index — no column changes, no data backfill, no `CASCADE DELETE` touched.
- **Migration is a DB-mutating action** — per `CLAUDE.md`, running `pnpm --filter api run prisma:migrate` requires explicit `[y/n]` confirmation at `/implement` time, and must be applied to both `notes_dev` (`DATABASE_URL`) and `notes_test` (`TEST_DATABASE_URL`) per the existing dual-apply convention (SDS §2.2/§18).

## Data Flow (Service Layer)

- **`listNotes(prisma, userId, query: ListNotesQuery)`**: unchanged scoping (`{ userId, deletedAt: null }`) and pagination (`skip`/`take`), only the `orderBy` source changes — from a hardcoded literal to `SORT_ORDER_BY[query.sort]`. Return shape (`Page<Note>` via the existing local `toPage` helper) is unchanged.
- **`listTrash`**: fully unchanged — still takes `PaginationQuery`, still hardcodes `orderBy: { deletedAt: 'desc' }`.

## Risk Areas

1. **Query-schema split (`listNotesQuerySchema` vs. `paginationQuerySchema`)**: two schemas now exist for what looks like "the same" page/pageSize shape. Mitigated by `listNotesQuerySchema = paginationQuerySchema.extend(...)` (single source, no duplication) and by an explicit integration test proving `GET /notes/trash` ignores a stray `sort` param — so the split doesn't silently regress Trash behavior.
2. **`sort` value shape as a single enum string** (`field:direction`) rather than two params: matches the exact format already documented in SDS §4, so no ambiguity, but means adding a 5th sort option later requires extending one enum plus one lookup record — both a single small diff, not a design smell.
3. **`tagIds` silently ignored rather than rejected**: acceptable per this ticket's Non-Goals (Zod strips unrecognized keys by default), but worth flagging in review since a caller could reasonably expect a `400` for an unsupported filter. AB-1006 turns this into real, honored filtering — no contract change needed then either, since the key already existed in the wire format, just unused server-side until now.
4. **Index-only migration risk is low**: no data transformation, no lock contention beyond a standard `CREATE INDEX` on the `Note` table's current size (still empty schema in dev/test). Same dual-database discipline risk called out in every prior ticket's plan (forgetting to apply to `TEST_DATABASE_URL`) — not new, just repeated.
5. **Sort-by-`updatedAt` test seeding**: integration tests must actually cause `updatedAt` values to diverge from `createdAt` order (e.g., `PATCH` an older note after seeding) to meaningfully assert `updatedAt:desc`/`updatedAt:asc` — a test that only seeds distinct `createdAt` values would pass even with a broken `updatedAt` sort, since `@updatedAt` initializes equal to `createdAt` on creation.
