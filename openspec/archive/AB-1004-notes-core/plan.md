## Prisma Schema Changes

```prisma
model Note {
  id        String    @id @default(cuid())
  userId    String
  title     String
  body      Json
  bodyText  String    @default("")
  version   Int       @default(1)
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  deletedAt DateTime?

  user     User          @relation(fields: [userId], references: [id], onDelete: Cascade)
  versions NoteVersion[]

  @@index([userId, deletedAt])
  @@index([userId, createdAt])
}

model NoteVersion {
  id      String   @id @default(cuid())
  noteId  String
  version Int
  title   String
  body    Json
  savedAt DateTime @default(now())

  note Note @relation(fields: [noteId], references: [id], onDelete: Cascade)

  @@index([noteId, version])
  @@index([savedAt])
}
```

- `User` model gains `notes Note[]`.
- No physical deletes anywhere in the request path — `softDeleteNote` only ever does `prisma.note.update({ data: { deletedAt: new Date() } })`. The **only** `prisma.note.delete`/`deleteMany` call in the entire codebase lives in `lib/jobs/purgeNotes.ts`.
- `onDelete: Cascade` on `NoteVersion.note` is scoped to that one purge job's physical delete — it never fires from a user-facing action, so it does not bypass the Note soft-delete rule (AGENTS.md §9/§11).
- **Migration is a DB-mutating action** — per `CLAUDE.md` permission rules, running `pnpm --filter api run prisma:migrate` requires explicit `[y/n]` confirmation at `/implement` time, and must be run **twice**: once against `notes_dev` (default `.env`), once against `notes_test` (`DATABASE_URL` temporarily swapped to `TEST_DATABASE_URL`, matching the existing dual-apply convention from AB-1002/AB-1003, SDS §2.2/§18).

## Data Flow (Service Layer)

- **`createNote(prisma, userId, input)`**: computes `bodyText = extractPlainText(input.body)`, `prisma.note.create({ data: { userId, title, body, bodyText } })`. No `NoteVersion` row (version starts at 1 with nothing to snapshot from).
- **`getNote(prisma, userId, id)`**: `findFirst({ where: { id, userId, deletedAt: null } })`; `null` → throw `AppError(404, NOTE_NOT_FOUND)`.
- **`updateNote(prisma, userId, id, input)`**: `findFirst({ where: { id, userId, deletedAt: null } })` first (404 if missing/trashed) — then a single `prisma.$transaction` containing (1) `noteVersion.create` with the **pre-update** `title`/`body`/current `version` number, and (2) `note.update` setting the new `title`/`body`/`bodyText` (recomputed if `body` changed) and `version: { increment: 1 }`.
- **`softDeleteNote(prisma, userId, id)`**: scoped `updateMany({ where: { id, userId, deletedAt: null }, data: { deletedAt: new Date() } })`; `count === 0` → 404 (covers not-found, not-owned, and already-deleted in one check, matching Scenario 9).
- **`listNotes(prisma, userId, page, pageSize)`** / **`listTrash(...)`**: `findMany` + `count` scoped by `userId` and `deletedAt: null` / `deletedAt: { not: null }` respectively, ordered `createdAt desc` / `deletedAt desc`; both return the shared `Page<Note>` shape via one small internal `paginate()` helper (kept local to `notes.service.ts` — this is the "same mechanism" AB-1005 extends, not a new shared abstraction warranted yet).
- **`restoreNote(prisma, userId, id)`**: scoped `updateMany({ where: { id, userId, deletedAt: { not: null } }, data: { deletedAt: null } })`; `count === 0` → 404 (covers not-found, not-owned, not-currently-deleted, and already-purged — all indistinguishable from the caller's perspective, matching Scenario 13).

## Risk Areas

1. **SDS says "Prisma middleware" for `bodyText` sync, but `$use` was removed in Prisma 6.14.0** (installed version here is `6.19.3`) — its replacement, `$extends`, returns a differently-typed client that isn't assignable to the plain `PrismaClient` type every existing service function (`auth.service.ts`) is typed against. **Resolved with user**: compute `bodyText` inline inside `notes.service.ts`'s `createNote`/`updateNote`, no client extension introduced. This is a deliberate, documented deviation from the SDS's literal wording, not an oversight.
2. **Express route ordering**: `GET /notes/trash` must be registered before `GET /notes/:id` in `notes.router.ts`, or Express will try to look up a note with id `"trash"`. Covered explicitly by an integration test.
3. **TipTap plain-text extraction robustness**: a malformed or unexpectedly-shaped `body` must never throw and block a save — `extractPlainText` is defensive-by-construction and unit-tested against garbage input.
4. **`restoreNote`/`softDeleteNote` race conditions**: using scoped `updateMany` (rather than `findFirst` + `update`) closes the TOCTOU gap where two concurrent requests could otherwise both "succeed" against a row already mutated by the other.
5. **Dual-database migration discipline**: forgetting to run `prisma migrate dev` against `TEST_DATABASE_URL` as well as `DATABASE_URL` would make integration tests fail at collection time with a missing-table error — this is a repeat of an existing AB-1002/1003 operational step, not a new risk, but worth calling out again since it's manual today (SDS §2.2/§18 has no automated dual-apply script).
6. **Open question carried forward, not blocking**: whether `packages/shared/src/types.ts` should export a dedicated `NotePage` alias or note-consuming code should just use the existing generic `Page<Note>` directly. Low-risk either way; will default to reusing `Page<Note>` directly (no new type) unless review flags a reason not to.
